# page-foundry v3.0 plan

Status: approved direction, work tracked on the v3.0 milestone. Research inputs live in `plans/research/` (six deep-read reports plus a completeness critique, 2026-07-17). This document is the system of record for what v3.0 is and why; the issues are the system of record for execution.

## Why 3.0

Three diagnoses, each grounded in the research pass:

1. **The archetype system defeats its own rule.** Shared rule 4 says sections appear in the order the skeptical reader raises objections; every archetype then freezes one order into numbered slots. A hundred users produce a hundred pages with the same open (hero, proof strip) and the same close (FAQ, final CTA), recognizable at thumbnail size. The skill reproduces the uniformity it exists to prevent. (research/pf-archetypes.md)
2. **Orchestration is doctrine without evidence.** Exactly one companion (humanizer) has a real invocation contract. The two highest-differentiation companions, customer-research and marketing-psychology, leave zero artifacts; deep use and token gestures are indistinguishable in every output the user sees. (research/pf-orchestration.md)
3. **The handoff balance is inverted.** The package over-prescribes where design tools excel (signature element pre-decided, tokens closed, no-merge rules) and under-specifies what tools cannot know (voice rules for generated microcopy, an explicit fabrication ban, asset geometry, a return spec, gates 6 and 7). (research/pf-handoff.md)

Plus one opportunity: **impeccable** (pbakaus/impeccable, Apache 2.0) is a design-fluency skill with a 46-rule deterministic anti-pattern detector, a Stitch-format DESIGN.md writer, and deep craft references. It is the design-side complement to everything page-foundry does on the copy and conversion side.

## Empirical checks already run

- `detect.mjs --json` against our own live page ran clean out of the box and produced two true findings: the `01/02/03` numbered section markers (named AI editorial scaffold) and the 6px `border-left` callout accent (named "the most recognizable tell of AI-generated UIs"). The design-side mechanical gate thesis is proven, including against our own output. Findings need a per-property accept/ignore mechanism, because a deliberate brutalist choice is legitimate — but it must be a recorded choice.
- `@google/design.md` exists on npm at 0.3.0 (linter + exporter). The published spec has eight body sections (impeccable's internal six-section doctrine is a stale profile; we target the published spec).

## Workstream 1: Archetypes become conversion contracts

The conversion rules are the asset and are already constraint-shaped; the archetypes wrap them in frozen sequences. Keep every rule, delete the slot numbers.

- Replace each archetype's numbered section list with a six-block contract: **Goal** (unchanged), **Buyer entry states** (keyed to the Phase 1 brief; this input generates section order), **Jobs the page must do** (current sections rewritten as unordered obligations), **Proof requirements** (type plus placement constraints), **CTA policy** (unchanged; best-evidenced rules in the system), **Ordering constraints** (the ~10 real invariants: 5-second hero first; proof within one viewport of every CTA; heaviest proof immediately before the ask; mechanism before price; risk reversal at the moment of commitment; FAQ collects leftovers only; mobile order hits proof before any form; message match binds single-source traffic; no qualified reader disqualified; sequence follows this buyer's objection map).
- **Mapper becomes a contract compiler.** Inputs: the three existing questions plus buyer awareness state and price/decision weight. Output: a filled contract, not a template name. Straddling = contract merge (union of jobs, conversion owner sets the goal, strictest policy wins). It reads foundry-log.md so per-property conversion data adjusts axis defaults.
- **Composition axes declared per run** in the Phase 2 spec: narrative shape (problem-led / demo-led / proof-led / offer-led, chosen by awareness state), hero form, proof strategy (concentrated vs interleaved), density (driven by price/decision weight).
- **Section-shape lexicon**: for each job, 3+ sanctioned shapes (proof: logo strip / inline pull-quote / case card / stat wall; how-it-works: steps / annotated screenshot / clip; objections: FAQ / inline callouts / not-for list / comparison table). Phase 2 picks jobs and order; Phase 4 picks shapes.
- **Multi-skeleton generation**: Phase 2 produces 2-3 constraint-satisfying candidate skeletons from the same contract, quick MECLABS pre-read on each, pick lands at the existing spec sign-off pause.
- **Anti-template check**: foundry-log gains a skeleton line (job order + axes); a new page whose skeleton matches the property's last N pages without conversion data justifying it gets flagged at Gate 1. Converged-because-it-converts is fine; converged-because-the-reference-said-so is the defect.

**New archetypes** (as contracts, ranked): pricing-page, comparison-alternatives, docs-dev-tool-landing, waitlist-coming-soon, thank-you-post-conversion (promoted from newsletter-capture's appendix; every archetype references it), event-webinar, agency-services, ecommerce-product, changelog-launch-post. 404-with-a-job ships in shared rules, not as an archetype. Portfolio is an axis setting on personal-home, which is evidence the contract model is right.

## Workstream 2: Evidence-based orchestration

Model everything on the humanizer contract, the one that works.

- **Companion table gains four columns**: INPUT (exact files handed over), OUTPUT (artifact path + required sections), CONSUMER (the phase that reads it), EVIDENCE (the gate-report line it fills).
- **Run artifact directory** `.agents/foundry/<product>/`: `voc.md`, `persuasion-map.md`, `message-architecture.md`, `page-spec.md`, `conversion-audit.md`, `tokens/`, `copy/`. Phases read files, not lingering context; runs become diffable and resumable.
- **Per-companion evidence lines** in the gate report: invoked / artifact written / what changed, or skipped (degraded). Rubber-stamping becomes visible.
- **VOC provenance schema**: every quote in voc.md carries a source. Gate 2 quote-tracing and Gate 8 integrity become mechanical checks against the file.
- **Independent conversion audit**: Gate 1's MECLABS score comes from a fresh subagent that receives only the rendered page and the brief, never the drafting context. Record divergence from the builder's self-score.
- **Multi-candidate beyond headlines**: hero-section candidates against different objection framings; frontend-design always produces 2-3 token plans; selections log reasons to the artifact directory.
- **Phase 3 ordering fix** (real defect): humanize, then cut, then red-team currently leaves post-humanizer edits unscanned until Gate 6. Any edit after the pattern pass re-triggers scan + humanizer. copy-editing outputs a changelog; the redundant only-cuts pass collapses into it.
- **Verbatim-copy integrity in build mode**: snapshot approved copy at the end of Phase 3; diff rendered text after any builder (web-artifacts-builder, /design-html) runs; drift fails Gate 2. Handoff mode has this; build mode, the default, does not.
- **Timing fixes**: ai-seo's extractable-answer-block constraint and analytics' conversion event move into the Phase 2 spec (gates verify, not retrofit). customer-research runs during intake to pre-fill the interview. competitors feeds the Phase 0/1 competitive frame, not just comparison sections. theme-factory gets a real trigger (no brand assets and no DESIGN.md) or leaves the table. pricing's Phase 2 output must generate Gate 6's /pricing.md.

## Workstream 3: impeccable as a foundational companion

Division of authority: page-foundry owns brief, message, conversion, voice, integrity, measurement; impeccable owns visual craft depth and mechanical design detection. MECLABS stays the arbiter on conversion pages; impeccable's Nielsen scoring is advisory there.

- **Companion table entry**: pinned source pbakaus/impeccable, detection via skill paths and `npx impeccable`, version from SKILL.md frontmatter.
- **Context handoff, no double interview**: page-foundry writes `PRODUCT.md` from its own brief (register: brand, platform: web, CTA/belief-ladder/proof mapped into Conversion & proof, anti-references verbatim) before any impeccable command runs. Singleton policy: PRODUCT.md/DESIGN.md live in the product's own repo root; if one exists that page-foundry did not write, read and merge, never clobber.
- **Phase 4**: `/impeccable document` persists the property's DESIGN.md (published Stitch spec profile); `palette.mjs` seeds greenfield color. This arms the detector's design-system enforcement (off-palette colors become findings).
- **Design ship gate**: `detect.mjs --json` over built HTML/CSS as a mechanical design scan alongside the render review. Findings are floors, not verdicts (impeccable's own doctrine, adopted verbatim). Per-property accepted-findings file for deliberate choices, mirroring `.impeccable/config.json` ignores.
- **Optional invocations**: `critique` when a page needs a UX-heuristics read beyond conversion; `live` for in-browser variant picking during explore; `overdrive` for the rare cinematic hero. impeccable's edit-time hooks stay off during foundry runs (phase gates own enforcement); documented.
- **Degraded parity**: absorb the top 15 transplantable craft rules into references/design-direction.md with provenance stamps (spacing rhythm, type ramp, bolder reject-list, one focal point, color strategy commitment, brand-tinted neutrals, three micro-bans, motion duration table, scene sentence + named anchors, identity lock, squint test, real imagery or nothing, focus/hover completeness, content hardening minimums, CLS/LCP mechanics). Full craft/critique/live machinery stays invocation-only.
- **Security posture**: impeccable's Node scripts run only when the companion is installed; page-foundry bundles nothing new; SECURITY.md documents the boundary.

## Workstream 4: Handoff 3.0

Canonical files stay the source of truth; a generated design.md is a projection, never the replacement.

- **Manifest**: 00-master-prompt (five non-negotiables: verbatim copy, section order per the contract, CTA policy, no fabricated proof, criteria = done; plus an explicit creative license grant for layout, art direction, motion, and signature-element alternatives), 01-copy (with image-slot dimensions and aspect ratios), 02-page-spec (the compiled conversion contract), 03-design-direction (locked tokens for existing brands, propose-then-lock for new properties, signature element as anchor-not-mandate), 04-acceptance-criteria (fixed template, criterion-ID mapped to gate-ID, gates 6 and 7 included), 05-voice-rules (voice.md excerpt governing any microcopy the tool generates), 06-return-spec (what comes back, and the revision-request format when a post-build gate fails), assets/ (theme.css, manifest with per-file slot/dimensions/alt, llms.txt and schema.jsonld as files).
- **design.md projection**: generated deterministically from 00-05 against the published Stitch spec (eight sections, version alpha pinned), linted with `npx @google/design.md lint`, regenerated on any change, never hand-edited. Per-tool packaging: attachment-capable tools get the package, single-prompt tools get design.md, repo-based tools get the directory with 00 as entry.
- **The fabrication ban travels to the tool**: the single most important instruction to an external generator (no invented logos, testimonials, ratings, stats) enters 00 verbatim.
- Fix now, pre-3.0: handoff.md's stale gate numbering (calls integrity Gate 6; it is Gate 8).

## Workstream 5: Gate upgrades

Anti-template check (WS1), evidence lines (WS2), independent conversion audit (WS2), verbatim-copy diff (WS2), design detect scan (WS3), squint-test line in render review, focus/hover completeness and CLS/LCP mechanics as concrete teeth in gates 3 and 4. Doctrine line adopted from impeccable: a clean mechanical scan is a floor, never a verdict.

## Workstream 6: Migration and release

- Phase 0 reads v2.x artifacts (old foundry-logs, tokens, specs) and adopts them into the artifact directory; no orphaned state.
- TESTS.md entries per workstream; regression tests for the contract compiler, evidence lines, verbatim diff, detect gate.
- Version 3.0.0 released in lockstep across npm, skills CLI, plugin, and .skill; release script keeps package.json, skill version, tag, and release synchronized and publishes only after TESTS pass.
- README, homepage, and llms.txt updated to describe contracts and the impeccable integration honestly (only after the features exist; no forward-looking claims on the page).

## Sequencing

1. WS1 (contracts + compiler + axes; the enabling change everything else references)
2. WS2 (artifact directory + evidence; makes WS1's outputs durable)
3. WS3 (impeccable; lands cleanly once Phase 4 reads/writes artifacts)
4. WS4 (handoff 3.0; consumes contract + artifacts + DESIGN.md)
5. WS5 woven through 1-4; WS6 last.

New archetypes (WS1's second half) can land incrementally after the contract schema exists.

## Decisions taken (so issues do not relitigate)

- MECLABS remains the conversion arbiter; impeccable critique is advisory on conversion pages.
- Target the published design.md spec (eight sections), not impeccable's internal six-section profile.
- PRODUCT.md/DESIGN.md singleton policy: product repo root, merge never clobber.
- Detector findings are floors; per-property accepted-findings make deliberate style choices recorded choices.
- Canonical handoff files are the source of truth; design.md is a generated projection.
- impeccable's hooks stay off during foundry runs; phase gates own enforcement.
