- **11g: anti-template check window set to the last three runs.** The plan says "last N pages" without picking N. Chose three: one repeat can be a coincidence of similar buyers, and a longer window would flag pages whose old skeletons predate the current contract set. The check is cheap either way (it reads three log lines), and the resolution is a justification in the spec, not a forced rebuild, so a too-tight N costs a sentence, not a page. Worth a morning glance if the owner wants a different window.
- **13a: core-tier hard stop applies to missing, not outdated.** The owner decision says preflight "stops until installed"; an outdated core companion is installed and still runs as the primary path, so declining its update proceeds (with the version gap noted), consistent with TESTS.md #10 ("outdated is offered, not forced"). Treating outdated-core as a hard stop would have contradicted an existing test to enforce a reading the decision text does not require.
- **13a: skill-package README trued immediately, root README deferred to 19b.** The two-tier policy made the skill README's "all optional" line false, and that file ships inside the package, so it was corrected in the same commit. The root README's "Built on" credit to theme-factory and its "all of these are optional" sentence are also now stale, but that file is repo marketing, owned by the 19b docs truth pass; deferred there rather than expanding 13a.
- **13b: the brief does not move into the run artifact directory.** The backlog line says Phase 0 "adopts v2.x artifacts (old .agents/product-marketing.md, foundry-log)", which could read as relocating the brief into `.agents/foundry/<product>/`. Chosen reading: the brief stays at `.agents/product-marketing.md` in the repo case, because that path is the marketingskills convention every core companion reads (product-marketing's own doc says all other skills reference it there); moving it would quietly break the core-tier orchestration 13a made mandatory. Only no-repo products keep the brief inside the foundry directory, replacing v2.x's vaguer "beside the page output". Adoption of the brief therefore means finding it where v2.x left it and confirming its v3 home, not relocating it. Worth a morning glance since the plan's directory listing is ambiguous on this.
- **11d: newsletter-capture gained a hero-proof mandate the v2.x file never had.** The v2.x numbered version (form in hero at slot 1, proof at slot 3) violated rule 8 / shared constraint 7 (mobile reader hits proof before any form) on every mobile render; the contradiction predates v3 but the contract conversion forced a resolution. Chosen fix: the contract's first ordering constraint requires a proof element in the same hero viewport as the form (creditable count, reader quote, or taste-it link beside the field), which satisfies shared constraints 2 and 7 instead of silently subtracting from the shared set (which the schema forbids). The alternative, carving a newsletter exception out of shared constraint 7, would have re-legalized the exact form-before-proof pattern rule 8 exists to prevent. Judgment call worth a morning glance: it adds a small design obligation to every newsletter hero.
