No fixture content; the product and its proof limits are real.

## Axis recommendations carried in from the compiler

- Narrative shape: demo-led. Solution-aware readers own the problem; the pipeline doing its job is the argument.
- Hero form: install-command. Pinned by the oss-project contract; not a per-run choice.
- Proof strategy: interleaved. The inventory has no heavy artifacts to concentrate (no claimable stars or adopters); the real output rides beside the claims.
- Density: compressed. Free install, minutes to try; the lightest weight on the gradient.

## Candidate A · demo-led, interleaved proof (recommended)

Order: hero with install command → quickstart with real output → how it works (gate list, terse) → honest comparison → project alive → close (real gate-report card beside the install repeat).

Constraint check: hero first with install in it (shared 1, oss); quickstart begins one scroll down, serving the product-aware minority (oss); comparison sits after the reader has seen it work (oss); the gate-report card at the close keeps proof within a viewport of the final CTA (shared 2) and puts the heaviest available proof immediately before the last ask (shared 3); mechanism-before-price and risk-reversal do not bind on a free install.

Pre-read: strongest clarity-to-friction ratio of the three; the demo answers "does it work" before the comparison asks "why this one", which is the order this buyer raises those objections. Leak: the how-it-works section has to stay terse or the compressed density promise breaks.

## Candidate B · demo-led, comparison early

Order: hero with install command → quickstart → honest comparison → how it works → project alive → close (gate-report card, install repeat).

Same axes as A; the difference is a genuinely different legal order. The objection map supports it: a solution-aware reader arrives holding alternatives (prompt tricks, other skills, a hired copywriter), so "why this over those" can land the moment the quickstart proves the thing runs.

Constraint check: identical to A except comparison position; still after first sight of it working (oss), so legal.

Pre-read: wins if most traffic arrives mid-comparison; leaks credibility where A does not, because conceded tradeoffs read stronger after the mechanism has built trust, and here they land on one screen of demonstration.

## Candidate C · proof-led, concentrated

Order: hero with install command → what it is and how it works, one viewport → quickstart → project alive → the output (a real page this skill built, with its real gate report; the heaviest proof, concentrated) → close (install repeat).

Different narrative shape and different proof strategy: the page argues from the finished artifact instead of the pipeline run.

Constraint check: the concentrated proof section sits immediately before the close, the cleanest satisfaction of shared 3 among the three; the how-it-works block must hold to one viewport so the quickstart still begins within one scroll for the product-aware minority (oss), which is tight on mobile and the spec would have to say so.

Pre-read: strongest final impression; two leaks. The demonstration arriving mid-page undercuts the practitioner instinct that the code is the proof, and a concentrated section built on a thin inventory risks exactly what the contract forbids: proof run thin instead of cut.

## Recommendation

A. It matches the compiled axis settings without any conditional-constraint tension, and its one leak is a discipline note rather than a structural cost. B is the live alternative if the traffic answer shifts toward readers arriving mid-comparison; C waits until the proof inventory has an artifact heavy enough to concentrate. Pick lands at spec sign-off.
